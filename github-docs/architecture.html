Technical deep dive

The tamux runtime architecture

tamux is a daemon-first multiplexer with an agent runtime built into the daemon. The key to understanding the system is seeing where state lives, how the agent executes work, and how durability, safety, and learning sit above the hot path.

1. System boundary

The important architectural boundary is that the daemon owns state and the UIs are clients of that state. That sounds simple, but it changes the entire product model. Sessions, goal runs, tasks, threads, approvals, memory, transcripts, and telemetry do not belong to the current browser window or terminal tab.

2. Main runtime components

Daemon

Owns PTY sessions, workspace state, agent threads, task queue, goal runs, approvals, persistence, telemetry, and memory.

Clients

Electron, TUI, CLI, MCP, and gateways are alternate operator surfaces over the same daemon-owned runtime.

Session manager

Handles PTY lifecycle, terminal input/output, history, transcript indexing, mission event rows, and snapshot metadata.

Agent runtime

Runs threaded conversations, tool execution, background tasks, goal planning, memory maintenance, health supervision, and learning loops.

3. Thread, task, and goal relationships

tamux separates short conversational turns from durable execution:

  • Threads hold persisted conversations and tool messages.
  • Tasks are daemon-owned units of work with status, priority, dependencies, retries, and session affinity.
  • Goal runs are the durable autonomy layer that plans, dispatches child tasks, monitors progress, replans, and reflects.
Practical rule: goal runs sit above the queue. They do not replace it. They use it.

4. The execution hot path

  1. A user or external client sends a message into a daemon thread.
  2. The daemon persists the message immediately.
  3. The runtime builds a prompt from system prompt, SOUL/MEMORY/USER, skills, operator model, operational context, and optional recall systems.
  4. The agent executes a bounded loop of model calls and tool calls.
  5. Tool results are persisted back into the thread.
  6. The assistant result is stored and broadcast to clients.
  7. Background learning, provenance, and health state are updated around the turn.

5. Self-orchestrating layers

  agent loop · dispatcher · tool executor · goal planner

Layer 0 — Execution

The hot path: LLM streaming, tool calls, child-task dispatch, goal-step progression, and direct runtime work.

Layer 1 — Liveness

Checkpointing, health monitoring, stuck detection, and recovery make long-running work durable instead of brittle.

Layer 2 — Meta-cognition

The runtime can assess its own progress, dynamically replan, escalate through multiple levels, and manage resource pressure.

Layer 3 — Learning

Execution traces, pattern mining, heuristics, and tool-effectiveness tracking help future runs make better decisions.

7. Why daemon ownership is a product advantage

  • Long-running work survives UI disconnects.
  • Multiple clients can attach to the same execution state.
  • Memory and tasks stay coherent across surfaces.
  • Approvals, audit trails, and run history stay central rather than fragmented.
  • Plugins and gateways can extend the same runtime instead of implementing separate mini-runtimes.

Without this ownership model, tamux would collapse back into a fancy prompt shell. The daemon is what makes the system feel like an operating environment.
