Operator guides

Getting productive with tamux

These guides are aimed at the real operator loop: getting the daemon running, choosing the right client surface, launching your first durable goal, and keeping long-running work reliable instead of vague.

1. Installation paths

Quick install

Use the shell installer when you want native binaries without pulling in npm.

curl -fsSL https://raw.githubusercontent.com/mkurman/tamux/main/scripts/install.sh | sh
tamux --help

Source / development install

Use source builds when you want the full daemon + Electron + TUI development workflow.

git clone https://github.com/mkurman/tamux.git
cd tamux
./scripts/setup.sh --check --profile source

For source work you will typically want Rust, Node.js, npm, git, and uv. The daemon is the thing that matters most operationally: start it first and treat the rest of the surfaces as clients that reconnect to its state.

2. First-run checklist

  1. Run preflight checks with ./scripts/setup.sh --check --profile source.
  2. Start the daemon: cargo run --release --bin tamux-daemon.
  3. Open the client surface you want to use first: Electron, TUI, CLI, or MCP.
  4. Confirm the runtime is pointed at the daemon backend.
  5. Inspect the goal, task, and approval surfaces exposed by your current client to verify the runtime is healthy.
First-run truth: if the daemon is not up, the rest of the system is missing its source of truth.

3. Running your first goal

The easiest way to understand tamux is to run a bounded, useful long objective instead of a casual chat turn. A good first goal is something concrete enough to verify, but large enough to benefit from planning, child tasks, approvals, and reflection.

Investigate why the nightly Rust build is failing, identify the root cause,
propose the smallest safe fix, and capture any reusable workflow as a skill.

What you should see

  • The run enters queued, then planning.
  • The daemon generates a plan and child tasks.
  • Risky operations pause behind approvals instead of silently proceeding.
  • The run ends with a reflection, and possibly a durable memory update or generated skill.

4. Choosing the right surface

SurfaceBest forTypical use
ElectronBroad desktop overview and settings controlGoal-oriented views, settings, plugin UI, and multi-pane visibility (surface area varies by build)
TUIKeyboard-first control and SSH-friendly workInspect sessions, goals, tasks, approvals, threads from a terminal
CLIScripting and direct inspectionAutomation, session management, health checks
MCPExternal agent integrationExpose tamux capabilities to outside tools or agent environments
GatewayChat-platform entry pointsDiscord, Slack, Telegram, or WhatsApp driven work

You do not need to pick one surface forever. The strength of tamux is that these are all clients of the same daemon state, even though feature depth is not yet perfectly identical across them.

5. Good daily habits

Treat threads as durable work

Keep one thread per objective. Continue it when the mission is the same. Start a new one only when the objective changes materially.

Use goal runners for real work

If the task is multi-step, long-running, or needs recovery and approvals, use a goal run instead of babysitting a single chat thread.

Be explicit about success

Tell the runtime what counts as done, what constraints matter, and what verification you expect. tamux rewards operational specificity.

Let the daemon own state

Do not treat the UI closing as a failure. The daemon should keep the work alive and reconnectable.

6. Model role guidance

tamux works best when different built-in fires are configured for different jobs. A common anti-pattern is assigning the same expensive, high-reasoning model to every role. A better setup is role-specific:

  • Svarog: strongest model available for primary execution, synthesis, and goal runs.
  • Weles: strong review/governance model for compaction, skeptical review, and risky-path checks.
  • Rarog: cheap, fast, and tool-reliable for lightweight operator assistance.
Do not starve governance: Weles does not need the single strongest model, but it should still be strong enough to make reliable review judgments.

7. Scenario playbooks

Bug hunt in a repo

Use a goal runner. Ask for a root-cause analysis, smallest safe fix, and explicit local verification. Let approvals guard risky mutation steps.

Release prep

Use a thread for early discussion, then move to a durable goal when you need checklist execution, packaging, validation, and handoff continuity.

Long research task

Use goal runners when the task spans multiple searches, comparisons, artifacts, or follow-up branches. That is where reflection and memory become valuable.

Remote / SSH operation

Use the TUI as the control room, keep the daemon running remotely, and reconnect as needed. This is where the daemon-first model shines.

8. When not to overuse tamux

Use a simple chat turn when the task is short, disposable, and does not need queueing, goals, approvals, recovery, or durable artifacts. tamux is strongest when continuity matters.
