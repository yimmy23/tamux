<objective>
Build daemon-side plugin skill bundling and command dispatch infrastructure.

Purpose: When plugins are installed, their bundled YAML skills must be copied to the standard skills directory so the existing skill discovery system finds them automatically. Plugin commands declared in manifests must be interceptable as slash commands that route to the plugin's API endpoint.

Output: `skills.rs` (copy/remove skill files), `commands.rs` (registry + dispatch), IPC for command listing, agent_loop command interception.
</objective>

    - After creating `plugin_manager` and `agent` (AgentEngine), set the plugin_manager on the agent engine:
      - The agent is wrapped in Arc, so either: (a) set it before Arc wrapping, or (b) use interior mutability.
      - Preferred: set it on AgentEngine before `Arc::new(agent)`. Read the existing server.rs construction flow. The agent is created, then plugin_manager is created. If they're created in sequence and agent needs plugin_manager, reorder: create plugin_manager first, then pass to agent construction.
      - Add `plugin_manager: Option<Arc<crate::plugin::PluginManager>>` parameter to AgentEngine constructor, or add a builder method.

    **Update `crates/amux-daemon/src/agent/agent_loop.rs`:**
    - At the top of `send_message_inner`, before the user message is added to the thread, check if the content looks like a plugin command:
      - A plugin command matches the pattern: starts with `/`, contains a `.` before any space (e.g., `/gmail-calendar.inbox`, `/weather.forecast London`)
      - Create a pure function `fn parse_plugin_command(content: &str) -> Option<(&str, &str, &str)>` that returns `(full_command_key, remaining_args)` or similar
      - If content matches, check `self.plugin_manager` for a resolved command
      - If command found:
        1. Add the user message to the thread as normal (so it appears in chat history)
        2. Instead of calling the LLM, directly build a system message like: "Plugin command `/{plugin}.{command}` dispatched. {description}. The agent will call plugin:{plugin}:{endpoint} to fulfill this request."
        3. Then proceed with the LLM turn, but prepend a system instruction that tells the agent to use the plugin_api_call tool (or equivalent tool) with the plugin name, endpoint, and any args from the user message
      - If content matches plugin command pattern but command is not found, proceed normally (let LLM handle it -- the user might be referencing something else)

    The key insight: we do NOT bypass the LLM. Instead, we augment the message with a system hint: "The user invoked plugin command /gmail-calendar.inbox. Use the plugin API tool to call plugin 'gmail-calendar', endpoint 'list_events' with these args: {remaining text}." This lets the agent use its existing tool-calling loop naturally.

    Add a helper method to AgentEngine:
    ```rust
    async fn try_augment_plugin_command(&self, content: &str) -> Option<String> {
        let pm = self.plugin_manager.as_ref()?;
        let entry = pm.resolve_command(content).await?;
        let args = content[entry.command_key.len()..].trim();
        Some(format!(
            "[Plugin command: {}]\nThe user invoked plugin command `{}`. \
             Plugin: '{}'. Description: {}. \
             Call the plugin API endpoint '{}' for plugin '{}'{} to fulfill this request.",
            entry.command_key,
            entry.command_key,
            entry.plugin_name,
            entry.description,
            entry.api_endpoint.as_deref().unwrap_or("default"),
            entry.plugin_name,
            if args.is_empty() { String::new() } else { format!(" with arguments: {}", args) },
        ))
    }
    ```

    In `send_message_inner`, right after adding the user message to the thread and before the LLM call, if `try_augment_plugin_command` returns Some, inject it as a system message into the thread so the LLM sees the command context.
