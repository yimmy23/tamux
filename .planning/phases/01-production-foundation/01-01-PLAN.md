# D-13 deferral note:
# D-13 requires "retry with backoff for critical writes." This plan implements
# busy_timeout=5000 (SQLite waits up to 5s before returning SQLITE_BUSY) which
# is the primary mitigation. Explicit application-level retry-with-backoff is
# consciously deferred because:
#   1. tokio-rusqlite serializes all operations through a single background thread,
#      making SQLITE_BUSY near-impossible in normal operation (it would only occur
#      if an external process holds a lock on the same database file).
#   2. WAL mode already allows concurrent readers alongside a single writer.
#   3. The combination of WAL + busy_timeout + single-connection serialization
#      eliminates the contention scenario that retry-with-backoff is designed for.
# If SQLITE_BUSY is observed in production despite these mitigations, retry-with-
# backoff can be added as a targeted fix in a future phase.

<objective>
Migrate HistoryStore from synchronous open-per-call SQLite to a single async tokio-rusqlite 0.6.0 connection with WAL mode pragmas.

Purpose: This is the riskiest and largest change in Phase 1. The current pattern opens a new SQLite connection for every operation (65 calls to `open_connection()` in history.rs). Migrating to tokio-rusqlite creates a single background thread that serializes all DB access, enabling concurrent usage from multiple daemon subsystems without "database is locked" errors. WAL mode allows concurrent readers while a write is in progress.

Output: A fully async `HistoryStore` with `tokio_rusqlite::Connection` field, WAL pragmas applied on construction, all 71 public methods migrated to async `.call()` closures, and `open_connection()` removed. Integration test stubs created for WAL and async connection behavior.

    **The pattern for EVERY method is identical:**

    BEFORE:
    ```rust
    pub fn some_method(&self, arg: String) -> Result<Vec<Item>> {
        let connection = self.open_connection()?;
        let mut stmt = connection.prepare("SELECT ...")?;
        // ... synchronous rusqlite code ...

    ```

    AFTER:
    ```rust
    pub async fn some_method(&self, arg: String) -> Result<Vec<Item>> {
        let arg = arg.clone(); // Clone any non-Copy args before closure
        self.conn.call(move |conn| {
            let mut stmt = conn.prepare("SELECT ...")?;
            // ... identical synchronous rusqlite code inside closure ...
            Ok(items)
        }).await
    }
    ```

    **Rules for the migration:**

    1. Add `async` keyword to the method signature.
    2. Remove `let connection = self.open_connection()?;` line.
    3. Wrap the remaining method body in `self.conn.call(move |conn| { ... }).await`.
    4. Replace references to `connection` with `conn` (the closure parameter).
    5. For methods that use `&self` fields inside the closure (e.g., `self.skill_dir`, `self.worm_dir`, `self.telemetry_dir`), clone those fields BEFORE the closure and use the cloned values inside. This is required because `.call()` closures must be `'static + Send`. Example:
       ```rust
       pub async fn some_method_using_paths(&self) -> Result<()> {
           let skill_dir = self.skill_dir.clone();
           let worm_dir = self.worm_dir.clone();
           self.conn.call(move |conn| {
               // Use skill_dir and worm_dir (owned), NOT &self.skill_dir
               Ok(())
           }).await
       }
       ```
    6. For methods that call other HistoryStore methods internally (helper chains), inline the SQL or call `.call()` directly -- do NOT call `self.other_method().await` inside a `.call()` closure (closures cannot be async).
